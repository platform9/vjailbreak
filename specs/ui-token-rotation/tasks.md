## Phase 4: User Story 2 — Automatic Token Renewal (Priority: P2)

**Goal**: Token renews automatically via kubelet projected volume. No pod restart needed ever.

**Independent Test**: `kubectl exec <ui-pod> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token | cut -d'.' -f2 | base64 -d` → verify `exp` claim is set and `exp - iat ≤ 86400`. For TTL test: set `expirationSeconds: 600`, wait 8+ minutes, verify token file content changes while API calls continue to succeed.

**Note**: Kubelet handles renewal at ~80% of TTL (≈ 19.2h for 24h default). No code needed beyond T002. This task locks k3s to respect the configured TTL.

- [ ] T015 [US2] Add `service-account-extend-token-expiration: false` and `secrets-encryption: true` to `image_builder/configs/k3s.config.yaml` — prevents k3s from silently extending projected token TTLs beyond the configured `expirationSeconds`

**Checkpoint**: Token `exp` claim matches configured TTL. k3s will not extend it indefinitely.

## Phase 5: User Story 3 — Configurable Token Lifetime (Priority: P3)

**Goal**: Single documented value controls token TTL. Operator changes it without touching source code.

**Independent Test**: Change `expirationSeconds` in `ui/deploy/ui.yaml` → redeploy → decode token → verify `exp - iat` matches new value (Kubernetes enforces 600s minimum regardless).

**Note**: US3 is fully served by T002 (`expirationSeconds: 86400`). This task adds discoverable documentation at the config entry point.

- [ ] T016 [US3] In `ui/deploy/ui.yaml`, add inline YAML comment to `expirationSeconds`: `# Operator-configurable: set to desired TTL in seconds (Kubernetes minimum: 600s; k3s extend-token-expiration must be false)`
